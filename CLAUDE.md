# れんしゅうメニュー ビルダー（球技）引き継ぎ書

## プロジェクト概要
特別支援学校（高等部・準ずる教育課程）の体育で、生徒が**自分で練習メニューを組み立てる**ためのWebアプリ。
体育原理（体育理論）の「目的→手段」の考え方に沿い、**①もくてき → ②ないよう → ③回数・セット・時間 → ④気をつけること** の順で1練習ずつ作り、今日のメニューとして積み上げる。

- 主対象：車いすバスケットボールを行う生徒（他の球技でも使える汎用設計）
- 表記はかみ砕いた日本語（ひらがな多め）。学習指導要領（高校保健体育・球技）のねらいを選択肢の参考にしている
- 開発者は非エンジニアの教員。**説明はかみ砕いて、修正は差分中心で**

## 技術スタック・構成
- **アプリ本体は index.html 1枚**（HTML+CSS+JS すべて。ビルドなし・外部ライブラリなし・Vanilla JS）
  - 2026-07-17 にホーム画面対応でアイコンと manifest を追加したため、**厳密な「単一ファイル」ではなくなった**。
    ただしロジックは引き続き index.html だけに閉じている
- ホスティング：**GitHub Pages**（リポジトリ：https://github.com/konkora780-ux/practiceplan ）
  - 公開URL：https://konkora780-ux.github.io/practiceplan/
- フォント：Google Fonts（Zen Maru Gothic / Zen Kaku Gothic New）
- 保存：**localStorage**（サーバーなし）
- 主な利用端末：**iPad（Safari）**。縦・横両対応、印刷（🖨ボタン→window.print）あり
- 提出：**AirDrop**（Web Share API）。https でないと動かない

周囲の色をにじませて埋めてから各サイズに縮小している（Pillow で加工）。

## 画面構成（上から順）
1. ヘッダー：日付（📅）と選択中の競技を表示
2. 競技グリッド：11競技＋自由（`SPORTS` のキー順に表示）
3. ステップマップ：1〜4の流れを先に見せる（メニューが1つでもあると全て✓表示）
4. ビルダー（4枚のカード）：
   - ①もくてき（単一選択チップ＋自由入力。自由入力が優先）
   - ②ないよう（単一選択チップ＋自由入力）
   - ③回数・セット・分（＋−ステッパー。0は「使わない」）
   - ④気をつけること（**複数選択**チップ＋自由入力の追記）
5. 「＋この練習をメニューに入れる」→ 今日のメニューに追加
6. 今日のメニュー：練習カードの一覧（下記の機能つき）
7. 今日のふりかえり：textarea（inputイベントで自動保存）
8. 印刷／今日のメニューを消す
9. 📚過去のメニュー：日付ごとの履歴（見る／今日にコピー／けす）
10. タイマーバー：画面下固定（`#timerBar`）。タイマー起動中のみ表示

## データ構造（重要）

### SPORTS（競技マスタ・JS内の定数）

```js
const SPORTS = {
  wcb: {
    name: "車いすバスケットボール",
    icon: "🏀",
    goals:  [ {t:"こうげき", items:["空いた場所を見つけて攻める", ...]}, ... ],
    drills: [ {t:"チェアワーク", items:["ダブルプッシュ", ...]}, ... ],
    cares:  [ {t:"安全", items:["転倒・接触に気をつける", ...]}, ... ],
  },
  ...
  free: { goals:[], drills:[], cares:["...", ...] }  // フラット配列も許容
};
```

- goals/drills/cares は **グループ形式 `{t:見出し, items:[...]}` と 文字列のフラット配列の両方に対応**（`fillChips` が typeof で分岐）
- 競技を追加するときは `SPORTS` に1ブロック足すだけ
- 競技キー：wcb / basket / soccer / handball / volley / tabletennis / wc_tabletennis / badminton / wc_badminton / tennis / softball / free

### 保存データ（localStorage）

| キー | 内容 |
|---|---|
| `pmb3_days` | `{ "YYYY-MM-DD": { menu:[練習配列], note:"ふりかえり" }, ... }` |
| `pmb3_sport` | 最後に選んだ競技キー |
| `pmb3_name` | 提出用のなまえ（端末ごとに1回入れれば済む） |

> **重要**：iPad Safari の ITP により、**7日間サイトに触らないと localStorage が消される**。
> 体育が週1回だとちょうど境目に当たるため、**必ずホーム画面に追加して使う**こと
> （ホーム画面から開いた Web アプリは ITP の7日ルールの対象外）。

| `pmb3_pending` | 送れなかった提出（最大30件。起動時・online復帰時・次の提出時に自動再送） |

### Supabase（提出の記録）

- **drops プロジェクトに相乗り**（ref=`hortngwcijhjagwmcoxo` / https://hortngwcijhjagwmcoxo.supabase.co ）。
  無料プランは**2プロジェクトまで**で drops と hoken で埋まっており、新規作成できなかったため。
  RLS はテーブルごとにかかるので同居しても安全（下記）。分離したくなったらテーブルごと移せる
- テーブル `public.submissions`。定義とポリシーは **`supabase.sql`**（SQL Editor に貼って実行ずみ）
- 列：`id / submitted_at / day / name / sport / menu(jsonb) / note / total_min / item_count`
- **RLS の考え方（ここを崩さないこと）**
  - anon キーは index.html に**べた書き**＝世界中から見える。生徒30台にキーを配って回れないため
  - だから **anon は INSERT のみ**。SELECT を足すと**キーを見た誰でも全生徒の記録を読める**
  - 先生は Supabase の画面（service_role が RLS を迂回）で見る
  - `authenticated` の SELECT ポリシーは作成ずみ＝先生用ページを作るときに効く
- **secret キー（`sb_secret_...`）は絶対にアプリに入れない**。`D:\Claudプロジェクト\drops\.env` にある
- 1件あたりの大きさは check 制約で制限（なまえ30字／note 2000字 等）。
  anon で INSERT できる＝理屈上は誰でもゴミを投げられるため、被害を抑える意図
- REST は**1回1000件まで**（drops で踏んだ制限）。先生用ページを作るときは offset でページング

練習1件のオブジェクト：

```js
{ sport:"wcb", goal:"...", drill:"...", reps:10, sets:3, mins:0,
  cares:["...",...], status:"" }   // status: "" | "good"(できた) | "hard"(むずかしかった)
```

- 今日の日付キーは `todayKey()`（端末ローカル時刻ベース）
- 旧バージョンのキー（pmb_ball_menu 等）は廃止済み。移行処理は無し（運用前のため不要と合意済み）

## 実装済み機能
- 競技切り替え（チップの中身が競技ごとに切り替わる）
- グループ見出し付きチップ（▼こうげき 等）。もくてき・ないよう＝単一選択、気をつけること＝複数選択
- メニュー追加・並べ替え（▲▼）・削除・今日の全消去（過去は残る）
- **実施チェック**：✓できた（緑）／△むずかしかった（赤）。トグル式。カード左帯と背景色が変わる
- **ふりかえりメモ**：日付ごとに自動保存
- **日付別履歴**：翌日以降「過去のメニュー」に自動表示。展開表示・今日へのコピー（status はリセット）・削除
- **合計時間**：mins>0 の練習の合計をヘッダー横に表示（回数・セットのみの練習は合計に含めない仕様）
- **タイマー**：mins>0 の練習に「⏱○分タイマー」ボタン。画面下バーに残り時間、終了時 WebAudio でビープ3回
- 印刷最適化：ビルダー・過去履歴・操作ボタンを非表示、チェック結果とふりかえりは印刷に含む
- XSS対策：ユーザー入力は `esc()` でエスケープしてから innerHTML に挿入
- **AirDrop 提出**：「先生に提出する」カード（なまえ入力＋提出ボタン）。`submitText()` が
  テキストを組み立て、Web Share API で共有シートを開く。ファイル名は
  `日付_なまえ_れんしゅうメニュー.txt`。**File を渡すのが要点**（テキストだけ共有すると
  受け取り側でメモに入り、誰の分か分からなくなる）。共有シート非対応の環境では
  ファイル保存にフォールバック（このときテキスト共有経路に落ちるとメモ行きになる）
- **ホーム画面対応**：apple-touch-icon / apple-mobile-web-app-capable / manifest。
  ホーム画面での表示名は「体育プラン」

| 関数 | 役割 |

## 確認ずみ（2026-07-17）
- **公開サイトでの動作**（https://konkora780-ux.github.io/practiceplan/ ）
  … 競技12件・チップ描画・提出カード・アイコン読み込み・JSエラーなし
- **ホーム画面対応は iPad 実機で検証ずみ**
  1. Safari で開くと案内バナーが出る
  2. ホーム画面に追加するとアイコンと「体育プラン」が正しく出る
  3. **ホーム画面から開くとバナーが消える**
     → standalone 判定が効いている＝**ITPの7日ルール対策が成立している**
- バナーの表示条件はコード上7パターンで検証ずみ
  （iPad Safari / ホーム画面起動 / iPadOS 13+ の UA が Mac になるケース / iPhone / Windows / Mac）
- **提出は iPad 実機で通しずみ**：提出ボタン → 共有シートが出る／同時に Supabase にも行が入る
- **RLS は実際に破ろうとして確認ずみ**（anonキーで SELECT/PATCH/DELETE を試行）
  - SELECT → `[]`。行があっても読めない
  - PATCH / DELETE → **HTTP 204 が返るが中身は変わらない・行も消えない**。
    RLS が対象行を隠すため「0件に成功」の扱いになるだけ。**204 を成功と誤読しないこと**
  - なまえ31字 → check 制約で 400
- ブラウザ（GitHub Pages のオリジン）から Supabase への CORS も通過ずみ
- 未送信の再送も検証ずみ（2件たまった状態から自動送信して成功）

## 未確認事項（最優先で実機確認）
1. タイマーの**ビープ音**：iOSの消音スイッチ・自動再生制限で鳴らない可能性。表示（残り時間・おわり！）は音と独立して動く
2. **日付をまたいだ保存**：翌日開いて過去メニューに移ることを確認
3. localStorage は**同一端末・同一ブラウザのみ**。端末を変えるとデータは引き継がれない（既知の仕様）
4. 車いす3競技のチップが長すぎて選びにくくないか（2026-07-17 の用語差し替え・増量後、実機未確認）
5. **AirDrop の行き先**：共有シートが出るところまでは確認ずみ。ただし**先生のiPadで受け取ると
   どこに入るか（「ファイル」アプリの想定）は未確認**。アプリを選ぶシートが先に出る可能性もある
6. 生徒の端末を実際に複数台使ったときの運用（同じ端末を使い回す場合、`pmb3_name` が
   前の生徒のまま残る。学級で1人1台なら問題ない）
